# Context Budgeting and Session Strategy

Use this workflow when Betty feels bloated, expensive, slow, or over-contextualized.

## Goal

Keep Betty strong without dragging huge irrelevant context through every task.

## Core idea

Betty should not solve every problem with one giant long-lived session.

Instead use three layers:

1. **light startup context** for identity and orientation
2. **targeted retrieval** for task-relevant files
3. **fresh helper sessions** for long, narrow, or code-heavy work

## Default stance

### Main Betty session

Use for:

- user interaction
- planning
- memory stewardship
- prioritisation
- delegation decisions
- final synthesis

### Helper sessions

Use for:

- deep coding
- focused research
- narrow investigations
- bounded drafting or refactoring
- work likely to consume lots of tokens

## Practical rules

- do not load the whole repo just because it exists
- prefer the smallest useful set of files
- avoid rereading startup files unless needed
- use project notes and workflow notes as targeted context
- write the durable conclusion back into the repo so future work is cheaper

## When to spawn a fresh helper

Prefer a fresh helper when the task is:

- long
- technical
- narrow in scope
- likely to involve many iterations
- likely to pull in lots of code or source material
- better treated as a specialist sub-problem

## When to stay in the main session

Stay in the main session when the task is:

- strategic
- personal
- memory-sensitive
- about identity, policy, or security stance
- mostly synthesis over already-known material

## Monitoring

Check session status when needed.

Watch for:

- high context percentage
- rising compaction count
- repeated rereads of the same context
- sluggishness caused by too much session baggage

## Cost discipline

Good cheap pattern:

- Betty plans the work
- helper does the deep bounded task
- helper returns a concise result
- Betty stores the durable outcome in markdown

Bad expensive pattern:

- one giant session
- huge repeated context
- no durable notes
- same work rediscovered over and over

## Output rule

The helper should return:

- a concise result
- a patch/diff
- a focused summary
- or a small set of recommendations

Not a giant repeated brain dump.

## Relationship to memory

Token savings come from durable memory.

If an important result is only left in chat, the system pays again later.
If the result becomes a well-placed note, later tasks can load only that note.

## Success condition

Betty stays:

- coherent
- cheaper
- easier to steer
- less likely to bloat
- better at turning one-off work into reusable memory
