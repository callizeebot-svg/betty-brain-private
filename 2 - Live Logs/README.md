# Live Logs

## Purpose

This folder is the living memory stream of Betty Brain.

It is not just a dumping ground for notes.
It is where raw signal becomes durable continuity.

If `3 - Daily Journal/` is the reflective day-stream, `2 - Live Logs/` is the operational evidence trail.
Use Live Logs for what happened, changed, or was decided — not for every rough thought.

Live Logs should capture:

- meaningful actions
- assistant actions
- decisions
- observations
- research findings worth preserving
- reflections and pattern notes
- evidence that later feeds project, area, people, and identity pages

## Memory model

Betty should keep memory in layers:

### 1. Raw capture

Temporary incoming material lands in inboxes, project notes, or source files.

### 2. Live Logs

Important signal gets synthesized into append-oriented logs here.

### 3. Entity pages

Stable understanding gets reflected into:

- `People/`
- `5 - Projects/`
- `6 - Areas/`
- identity and aspiration docs

### 4. Review and dream-cycle maintenance

Recurring maintenance workflows should:

- strengthen links
- surface contradictions
- identify stale assumptions
- promote important patterns into more durable pages

## Core logs to maintain

### `ASSISTANT_ACTIONS_LOG.md`

Records important actions Betty took on Cally's behalf.

Use for:

- meaningful setup changes
- routing/delegation decisions
- project actions
- diagnostics and repairs
- recurring operational actions worth remembering

Do not log every tiny move.
Prefer meaningful actions with enough context to be useful later.

### `ACTIONS_LOG.md`

Records important user/system actions, events, and developments that shape ongoing work.

Use for:

- major decisions
- project changes
- environment changes
- important discoveries
- operational milestones

### Optional specialized logs later

If needed, Betty may later add:

- `DECISIONS_LOG.md`
- `REFLECTIONS_LOG.md`
- `RESEARCH_LOG.md`
- `INCIDENTS_LOG.md`

Only add them when the signal volume justifies the split.

## Writing rules

- append, do not silently rewrite past events
- keep entries concise but meaningful
- include dates
- preserve attribution when relevant
- link to related projects, areas, people, or operations docs where useful
- avoid secrets and auth-bearing content

## Dream-cycle relationship

Live Logs are the input stream for dream-cycle maintenance.

The dream cycle should review recent logs and:

- merge duplicate concepts
- surface patterns
- refresh project/area state
- identify contradictions
- suggest what belongs in longer-lived docs

This is how Betty gets stronger memory instead of just more notes.
