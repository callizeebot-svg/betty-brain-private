Executive summary

The strongest videos converge on one idea: serious agents need layered memory, not just longer prompts.

  • Consensus: split memory into short-term/session memory, long-term/persistent memory, and structured state/notes.
  • Strongest external ideas for Betty: context compaction, memory triggers, structured state injection, persistent retrieval, and safety/versioning around memory writes.
  • Betty Brain’s edge: user-owned markdown memory, git history, inspectable logs, human-readable structure, and an orchestrator/helper model that already thinks in durable memory.
  • Betty Brain’s main gap: the memory architecture is conceptually strong but still lighter on live retrieval automation, runtime memory tooling, and fully active recurring synthesis workflows.

Betty Brain strengths vs weaknesses

AreaStrengthWeakness / gap
Memory ownershipBetty stores durable memory in user-owned markdown files, not in opaque chat history.The content corpus is still early; the architecture is ahead of the actual memory density.
VersioningGit history is a real advantage. Few public agent-memory setups treat version control as first-class memory safety.Versioning is now documented well, but it still depends on disciplined execution rather than full automation.
Human inspectabilityObsidian + repo notes + logs make memory visible and editable.Manual structure can create friction compared with fully managed memory services.
Context cost controlBetty now has an explicit strategy for targeted retrieval and helper sessions.Live compaction and memory-trigger automation are not yet deeply wired into all flows.
Long-term memory modelThe layered model (journal, logs, projects, MEMORY, operations) is stronger than many single-store tutorials.Semantic retrieval/indexing is not yet a fully verified live path.
Short-term runtime memoryOpenClaw session context plus helper session strategy gives a workable short-term layer.It is not yet as explicitly tooled as the sessions/events/state patterns shown in Google ADK videos.
Safety and governanceBetty-governed root files and helper boundaries are a serious architectural advantage.Hard technical enforcement is still lighter than the policy/docs layer.
Skill/runtime wiringThe repo documents canonical skill logic and wrapper strategy.Some Betty-specific skills are documented but not fully wired into runtime discovery yet.

What Betty should copy next from this study

  1. Formalize session/state memory: make short-term runtime state more explicit, not just implied by chat/session history.
  2. Add retrieval triggers: decide when to search notes, when to summarize, and when to write back.
  3. Make compaction operational: convert “context budgeting” from guidance into active behavior.
  4. Strengthen long-term retrieval: wire a verified semantic index or equivalent retrieval path on top of the markdown brain.
  5. Upgrade memory quality control: add more systematic review loops for contradictions, stale facts, and note promotion.
